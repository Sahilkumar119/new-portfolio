SEC 01/ FOUNDATION

Basics: the everyday commands

Before you can do anything fancy with Git, you need the handful of commands you will run dozens of times a day.

Sahil is starting a small web app called taskboard, a simple board where people add and tick off tasks. He has not touched Git in about a year, so the muscle memory is gone. Rather than memorising a wall of commands, he is going to learn the daily loop: set things up once, save snapshots of his work, and look back when something breaks. Everything in this section happens on his own machine. Sharing work with teammates lives in a separate "Working with remotes" section later, so we will not touch that here.

1. First-time setup (you do this once per machine)

When Sahil makes a commit, Git stamps it with a name and email so the history shows who did what. If he never sets these, Git either complains or guesses badly. So the very first thing he does on a fresh machine is tell Git who he is. The --global flag means "remember this for every project on this computer," and Git stores it in a file in his home directory (~/.gitconfig).

terminal
$ git config --global user.name "Sahil"
$ git config --global user.email "sahil@example.com"

# Make new repos start on a branch called "main" instead of "master"
$ git config --global init.defaultBranch main

# Check what Git thinks is true right now
$ git config --global --list
user.name=Sahil
user.email=sahil@example.com
init.defaultBranch=main

There are two levels of config, and the distinction matters. Global config (--global) applies to every repository for this user. Local config (no flag, run inside a repo) applies only to that one project and lives in its .git/config file. Local always wins over global. This is useful when Sahil wants to use a different email for a particular project (for example, a work email on a work repo) without changing his default everywhere.

Tip

Run git config --list --show-origin when something looks wrong. It prints every setting and the exact file it came from, so you can see whether a value is coming from your global file or a local repo override.

2. Starting a repository: init versus clone

Sahil already has a taskboard folder with a few files in it, so he is not copying anyone else's project. He wants to turn his existing folder into a tracked repository. That is what git init does. If instead he wanted to grab an existing project that already lives somewhere else, he would use git clone to make a full local copy. He does not need clone here, so init is the right tool.

terminal
$ cd taskboard
$ git init
Initialized empty Git repository in /home/sahil/taskboard/.git/

$ ls -a
.  ..  .git  index.html  app.js  style.css

Notice the new .git folder. That hidden folder is the repository. It holds every commit, every branch, all the history, and the config for this project. Your actual files (index.html, app.js) sit next to it in plain view, but the entire memory of the project lives inside .git.

Watch out

If you delete the .git folder, you delete the whole history. The working files remain, but every commit, branch, and saved snapshot is gone permanently. Treat .git as the heart of the repo and do not remove it unless you genuinely mean to un-track the project.

3. The three areas (the mental model that makes Git click)

This is the single most important idea in this section, so let us tell it as a story. Sahil edits three files in one afternoon: he finishes a new "add task" button in app.js, he fixes a typo in index.html, and he starts some half-broken experimental styling in style.css that does not work yet. He wants to save the button and the typo fix together as one clean snapshot, but he absolutely does not want the broken styling in that snapshot. How does Git let him pick and choose? Through three areas.

The working tree is your folder as it exists right now, with all your latest edits, finished or not. The staging area (also called the index) is a holding pen where you place exactly the changes you want in your next commit. The repository is the permanent history of committed snapshots, stored inside .git. The flow always goes in one direction: working tree, then staging area, then repository.

The staging area is what trips most people up, so here is why it exists. It lets Sahil build a commit deliberately instead of dumping everything at once. He stages app.js and index.html, leaves the broken style.css alone, and commits. The result is one focused commit that says "add task button and fix heading typo," with the messy styling left out entirely. Good staging is the difference between a history you can read later and a pile of "fixed stuff" commits you will hate.

4. git status (the command you run most)

Before doing anything, Sahil checks where things stand. git status is the command he runs more than any other. It shows which files are changed, which are staged for the next commit, and which Git is not tracking at all. Run it constantly. It is read-only and never changes anything, so there is no risk in checking.

terminal
$ git status
On branch main

No commits yet

Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
        modified:   app.js
        modified:   index.html
        modified:   style.css

nothing added to commit but untracked files present

5. git add (choosing what goes into the snapshot)

Now Sahil stages exactly what belongs in this commit. He adds the two finished files by name and deliberately leaves the broken style.css in the working tree. There are a few ways to add, and each has a purpose.

git add app.jsStage one specific file.
git add app.js index.htmlStage several named files at once.
git add -AStage everything (new, modified, and deleted). Convenient, but it is easy to sweep in files you did not mean to.
git add -pStep through your changes hunk by hunk and choose yes or no for each piece, even within a single file.

The -p (patch) mode is the power tool. If Sahil accidentally made two unrelated changes inside app.js, git add -p lets him stage only the part that belongs in this commit and leave the rest for later. It is how you keep commits clean even when your editing was messy.

Watch out

The staging area is a snapshot of the file at the moment you ran git add, not a live link to it. If you stage app.js, then edit app.js again, those newer edits are not staged. You must run git add app.js a second time to include them. Running git status will show the same file as both "staged" and "not staged" when this happens, which is your clue.

6. git commit (saving the snapshot)

With the right files staged, Sahil records the snapshot into the repository. A commit needs a message describing what changed and why. The -m flag lets him write that message inline.

terminal
$ git add app.js index.html
$ git commit -m "Add task button and fix heading typo"
[main (root-commit) 9f2c1ab] Add task button and fix heading typo
 2 files changed, 18 insertions(+), 1 deletion(-)

# style.css is still sitting in the working tree, uncommitted, exactly as planned
$ git status
On branch main
Changes not staged for commit:
        modified:   style.css
git commit -m "message"Commit whatever is currently staged, with an inline message.
git commit -am "message"Stage and commit all already-tracked modified files in one step. It skips git add, but it does NOT include brand-new untracked files.
git commit --amendReplace the most recent commit with a new one (to fix the message or add a forgotten file).

The -am shortcut is handy once a file is already tracked, but remember its blind spot: a new file Git has never seen is untracked, and -am will silently ignore it. For new files you still need an explicit git add first.

Destructive

git commit --amend does not edit the last commit, it replaces it with a brand-new commit that has a different identity (a different hash). The old commit is discarded. This is perfectly safe for fixing a commit you have only made locally, which is exactly Sahil's situation here. It becomes genuinely dangerous once a commit has been shared with others, because you are rewriting history they already have. The rule of thumb: amend freely on commits that have never left your machine, and treat amend as off-limits for anything already shared.

7. Looking back: log, show, and diff

A week later something is off, and Sahil wants to retrace his steps. Three commands cover almost all of this. git log shows the history of commits, git show zooms into one specific commit, and git diff shows changes that have not been committed yet. All three are read-only, so explore freely.

git logFull commit history, newest first.
git log --onelineOne compact line per commit. The view you will use most.
git log --oneline --graph --allCompact history drawn as an ASCII graph, across all branches. Great for seeing how branches relate.
git show <commit>Show the message and the exact line changes for one commit.
git diffChanges in the working tree that are NOT yet staged.
git diff --stagedChanges that ARE staged, the exact diff your next commit will record.
git diff HEADEverything different from the last commit, staged and unstaged together.
terminal
$ git log --oneline --graph
* 9f2c1ab Add task button and fix heading typo

# What is different right now versus what I have staged?
$ git diff
diff --git a/style.css b/style.css
@@ -4,3 +4,5 @@
   color: #333;
+.task--done { opacity: 0.5; }   /* still experimenting */

# And what would actually go into my next commit if I committed now?
$ git diff --staged
(no output: nothing is staged yet)

The key distinction: plain git diff answers "what have I changed but not yet staged?" while git diff --staged answers "what exactly am I about to commit?" Getting comfortable reading both is what lets you commit with confidence instead of hope.

8. .gitignore (telling Git what to ignore)

Sahil's project soon grows a node_modules folder and some local log files. He never wants those in his history, and he is tired of seeing them clutter git status. A .gitignore file solves this. It is a plain text file, usually at the project root, listing patterns of files Git should pretend it does not see.

terminal
# .gitignore at the root of taskboard

# Dependencies
node_modules/

# Logs and local files
*.log
.env

# OS junk
.DS_Store

There is one rule that bites people constantly, so be clear on it: .gitignore only affects untracked files. If a file is already tracked (you committed it once before adding it to .gitignore), Git keeps tracking it and the ignore rule does nothing. To stop tracking something you already committed, you have to remove it from Git's tracking explicitly with git rm --cached <file>, and then commit that removal. The ignore rule alone is not enough.

Tip

Add your .gitignore early, ideally right after git init and before your first commit. That way junk like node_modules never gets tracked in the first place, and you avoid the "already tracked" headache entirely.

That is the complete local loop: configure once, init the repo, then edit, status, add, commit, and lean on log and diff to look back. Master this rhythm on your own machine first. Once it feels natural, the next section shows how branches let you work on several things at once without them colliding.

SEC 02/ BRANCHING

Branching: parallel lines of work

Sahil's app works. Now he wants to add a login feature, and he is terrified of breaking the working code while he experiments.

Here is the situation. Sahil has a project on the main branch that runs fine. He needs to build a new login feature, but that work will take days and involve half-broken code along the way. If he commits all of that directly onto main, anyone who pulls his code gets a broken app. He needs a parallel line of work that he can mess up freely, then merge in only when it is finished. That parallel line is a branch.

The key thing to understand, and the thing that confuses most people, is what a branch actually is. A branch is not a copy of your files. It is not a folder. A branch is simply a movable pointer to a single commit. That is the whole definition. When Sahil makes a new commit on a branch, Git moves the branch pointer forward to that new commit. Because a branch is just a tiny file containing one commit ID (40 hexadecimal characters), creating a branch is instant and nearly free. You are not copying the project, you are writing down one address.

Creating and listing branches

Sahil starts by seeing where he is and then creating a branch for his feature.

git branchList local branches. The current one is marked with an asterisk.
git branch feature/loginCreate a new branch pointing at the current commit. Does NOT switch to it.
git branch -aList all branches, including remote-tracking ones.
git branch -d feature/loginDelete a branch, but only if it has been merged. Safe.
git branch -D feature/loginForce-delete a branch even if unmerged. Destructive.
git branch -m old-name new-nameRename a branch.
terminal
$ git branch
* main
$ git branch feature/login
$ git branch
  feature/login
* main

Notice that after git branch feature/login, the asterisk is still on main. Creating a branch did not move Sahil onto it. Both branches now point at the exact same commit. Think of it as putting a second bookmark on the same page. Nothing has diverged yet.

Switching: git switch and git checkout

To actually start working on the feature, Sahil has to move onto the branch. There are two commands that can do this, and there is a history worth knowing.

For years, the one command git checkout did far too many different jobs. It switched branches, it restored individual files, it created detached states, and more. Overloading one verb with that many unrelated behaviors made it easy to do something you did not mean to do. So in Git 2.23 (2019) the maintainers split that behavior into two clearer commands: git switch for moving between branches, and git restore for throwing away changes to files. git checkout still works and is not deprecated, but for new learners git switch is the clearer mental model: it does one thing, it switches branches.

git switch feature/loginMove onto an existing branch.
git switch -c feature/loginCreate a new branch AND switch to it in one step.
git switch -Switch back to the previous branch you were on.
git checkout feature/loginOlder equivalent of git switch (still works).
git checkout -b feature/loginOlder equivalent of git switch -c.

The -c flag on switch (and the equivalent -b on checkout) is the one Sahil will use most. It collapses "create the branch" and "move onto it" into a single command, which is almost always what you want.

terminal
$ git switch -c feature/login
Switched to a new branch 'feature/login'
$ git branch
* feature/login
  main

Now the asterisk is on feature/login. Any commit Sahil makes lands here, and main stays frozen at the last good state. He can break things freely.

HEAD and the detached HEAD state

There is one more pointer to understand, because it explains an error message that scares everyone the first time they see it. That pointer is called HEAD. Normally HEAD points at a branch (for example, it points at feature/login), and that branch points at a commit. This is how Git knows "where you are" and which branch to move forward when you commit.

Sahil gets into trouble when he wants to inspect an old commit. Say he checks out a specific commit by its ID to see what the code looked like three commits ago.

terminal
$ git switch --detach 9f3c1a2
HEAD is now at 9f3c1a2 Add password hashing
$ git status
HEAD detached at 9f3c1a2

Now HEAD points directly at a commit instead of at a branch. This is "detached HEAD". It is not an error and nothing is broken. But it is a warning worth taking seriously, because any commits Sahil makes here are not attached to any branch. They float. If he then switches away to another branch, those floating commits have no name pointing to them, and Git will eventually garbage-collect them. The work is effectively lost unless he wrote down the commit ID.

Watch out

Commits made in detached HEAD belong to no branch. If you switch away without creating a branch first, those commits become unreachable and can be permanently deleted by Git's cleanup. To keep work made in this state, run git switch -c some-new-branch before leaving.

Getting out is simple. To discard the detour and go back to normal work, Sahil just switches to a real branch. To keep what he did, he creates a branch from where he is standing first.

  1. If you want to keep the work: run git switch -c feature/experiment to anchor these commits to a real branch.
  2. If you want to abandon the detour: run git switch main (or any branch name) to reattach HEAD and return to normal.

Tracking branches and upstream

So far everything Sahil did is local. To share the feature, he pushes it to a remote (for example, GitHub). The first time he pushes a new branch, he should set its "upstream", which links his local feature/login to the matching branch on the remote. Once that link exists, plain git push and git pull know where to go without him spelling out the remote and branch every time.

git push -u origin feature/loginPush the branch and set origin/feature/login as its upstream.
git branch -vvList branches showing their upstream and how far ahead/behind they are.
git branch --set-upstream-to=origin/feature/loginSet the upstream for the current branch without pushing.
terminal
$ git push -u origin feature/login
 * [new branch]      feature/login -> feature/login
branch 'feature/login' set up to track 'origin/feature/login'.
$ git branch -vv
* feature/login 4a1f9c0 [origin/feature/login] Add login form
  main          7b2e8d1 [origin/main] Initial setup

The -u in git push -u is shorthand for --set-upstream. After running it once, Sahil can just type git push from this branch and Git knows it means origin/feature/login. The [origin/feature/login] shown by git branch -vv is the proof that the tracking link is set.

Renaming a branch

Sahil realizes he named the branch poorly. Maybe he typed feature/logn with a typo. He renames it with -m (for "move").

terminal
$ git branch -m feature/logn feature/login
# rename the branch you are currently on (omit the old name):
$ git branch -m feature/login

Renaming a local branch is purely local. If the old name was already pushed to the remote, renaming locally does not rename it on the remote. Sahil would have to push the new name and delete the old remote branch separately.

Branch naming conventions

Branch names are just strings, so Git does not force any structure. But teams adopt prefixes so that a glance at the branch list tells you what each branch is for. The slash in names like feature/login is not a folder, it is just a convention that many Git tools display as a grouping.

feature/loginA new feature. Use feature/ for additive work.
fix/null-pointer-on-logoutA bug fix. Use fix/ (or bugfix/) for corrections.
hotfix/payment-downAn urgent fix going straight to production.
chore/update-depsMaintenance with no user-facing change.

Good names are lowercase, hyphen-separated, and short but descriptive. feature/user-profile-avatar tells the team exactly what is happening. feature/stuff tells them nothing.

Branch issues and gotchas

This is where beginners get burned, so read it carefully.

Switching with uncommitted changes. Suppose Sahil edited a file on feature/login but has not committed it, then tries to switch to main. Git's behavior depends on whether the change conflicts. If the file he edited differs between the two branches, Git refuses to switch and warns him, so he does not silently lose work. If the change does not conflict, Git carries the uncommitted edit along with him to the other branch, which can be surprising. The clean habit is to commit or stash before switching, so switching is always predictable.

terminal
$ git switch main
error: Your local changes to the following files would be overwritten by checkout:
        src/auth.js
Please commit your changes or stash them before you switch branches.
Aborting

Deleting a local branch versus a remote branch. These are two completely separate actions. git branch -d feature/login deletes only Sahil's local copy of the branch. The branch still exists on the remote and on every teammate's machine. To delete it on the remote, he needs a different command entirely.

git branch -d feature/loginDelete the LOCAL branch (if merged).
git push origin --delete feature/loginDelete the branch on the REMOTE.
git fetch --pruneRemove local references to remote branches that no longer exist.
terminal
$ git push origin --delete feature/login
 - [deleted]         feature/login
Destructive

git push origin --delete feature/login removes the branch from the shared remote for everyone. If that branch held work that was never merged and never copied elsewhere, it is gone. Confirm the branch is merged (or no longer needed) before deleting it on the remote.

Stale branches. Over time, branches pile up. After a feature is merged and the remote branch is deleted, Sahil's machine still remembers the old remote-tracking reference (origin/feature/login) until he prunes it. Running git fetch --prune (or git remote prune origin) cleans out references to remote branches that no longer exist, so git branch -a stops showing ghosts. Deleting merged local branches regularly, and pruning, keeps the branch list honest and readable, which matters more than it sounds when a project has been running for months.

SEC 03/ MERGING

Merging: bringing work back together

I'll write this section directly. The task is teaching content with strict HTML output, so I'll produce the inner HTML now.

Sahil's add-search branch is finished and tested. Now he needs that work to live in main where everyone else can see it.

Merging is the act of taking the commits from one branch and weaving them into another. Sahil has been committing happily on his feature branch while main kept moving (or maybe it sat still). His job now is to combine the two histories into one. Git gives him a few ways to do this, and the difference between them is not cosmetic. It changes the actual shape of your project's history, which is the thing you will stare at for years when you run git log trying to understand why something broke.

The mental model to hold onto: a branch is just a movable label pointing at one commit. main points at a commit, add-search points at another. Merging asks Git to reconcile those two pointers.

Fast-forward: when main never moved

Suppose Sahil branched off main, did his work, and nobody else touched main in the meantime. The history is a straight line. main is sitting at commit C, and add-search has carried on to E. There is no divergence, no competing work. In this case Git does the simplest possible thing: it slides the main label forward to point at E. That is a fast-forward. No new commit is created, because none is needed. The line was already straight.

terminal
before fast-forward:

A---B---C          (main)
         \
          D---E    (add-search)

$ git switch main
$ git merge add-search
Updating c3d4e5f..e5f6a7b
Fast-forward
 src/search.js | 42 ++++++++++++++++++++++++++++++++++++++++++
 1 file changed, 42 insertions(+)

after fast-forward:

A---B---C---D---E  (main, add-search)

Notice that after the fast-forward, both labels point at the same commit and the history shows no trace that a branch ever existed. That is fine for tiny changes, but it can erase the story of "these five commits were one feature." Sometimes you want that story preserved, which is where --no-ff comes in.

A true merge commit: when both branches moved

Now the realistic case. While Sahil was building search, a teammate merged a bugfix into main. The histories have diverged: main has a commit Sahil's branch does not have, and Sahil's branch has commits main does not have. There is no straight line to slide along anymore. Git cannot fast-forward. Instead it creates a brand new commit, called a merge commit, that has two parents: the tip of main and the tip of add-search. This commit is the knot that ties the two histories together.

terminal
before merge (diverged):

A---B---C---F        (main)   <- teammate's bugfix F
         \
          D---E      (add-search)

$ git switch main
$ git merge add-search
Merge made by the 'recursive' strategy.
 src/search.js | 42 ++++++++++++++++++++++++++++++++++++++++++
 1 file changed, 42 insertions(+)

after merge (M has two parents, F and E):

A---B---C---F---M    (main)
         \     /
          D---E      (add-search)

If the same lines were edited on both sides, Git stops and reports a merge conflict instead of finishing. It cannot guess whose version wins, so it hands the decision to you. You edit the conflicted files, run git add on them, and then git commit to complete the merge. Conflicts are normal, not a sign you did something wrong.

Forcing a merge commit with --no-ff

Recall the fast-forward case erased the fact that a feature existed as a group of commits. Many teams dislike that, because it makes it hard to see and to revert a feature as a unit. The flag --no-ff tells Git: even if you could fast-forward, do not. Create a merge commit anyway. The result is a visible bubble in the graph that says "this clump of commits was one feature, merged on this date." It also gives you a single commit to revert if the whole feature turns out to be a mistake.

terminal
$ git switch main
$ git merge --no-ff add-search
Merge made by the 'recursive' strategy.

A---B---C-------M    (main)
         \     /
          D---E      (add-search)

Use --no-ff when you care about traceability of features, which is most of the time on a team. Use a plain fast-forward when the change is trivial and you do not want noise in the graph.

Rebase: the other way to bring work together

Merging preserves exactly what happened, including the fact that two branches existed in parallel. Rebasing tells a different story. Instead of tying histories together with a merge commit, git rebase picks up Sahil's commits, sets them aside, moves his branch's starting point to the current tip of main, and then replays his commits one by one on top. The result is a straight line again, as if Sahil had started his work from the latest main all along.

The key word is replays. Rebase does not move your original commits. It creates new commits with the same changes but different parents, which means they get new commit hashes. Your old commits are abandoned (Git keeps them in the reflog for a while, but they are no longer on your branch). This is exactly why rebasing rewrites history.

terminal
before rebase (diverged):

A---B---C---F        (main)
         \
          D---E      (add-search)

$ git switch add-search
$ git rebase main
Successfully rebased and updated refs/heads/add-search.

after rebase (D and E replayed as D' and E' on top of F):

A---B---C---F---D'---E'   (add-search)
            ^
            (main still at F until you merge)

The typical workflow Sahil would use: from his feature branch, run git rebase main to replant his work on top of the latest main, resolve any conflicts as the commits replay, then switch to main and do a fast-forward merge. The end result is a perfectly linear history with no merge bubbles, which many people find far easier to read.

Watch out

The golden rule of rebasing: never rebase commits that you have already pushed and that other people may have pulled. Because rebase replaces commits with new ones (new hashes), anyone who based work on the old commits now has a diverging history. Their next pull turns into a tangle, and someone usually "fixes" it with a force push that destroys a teammate's work. Rebase freely on your own local, unpushed branch. Once it is shared, merge instead.

Merge versus rebase: the honest trade-off

This is a genuine trade-off, not a case where one is simply correct. Merge keeps history truthful and non-destructive: it records exactly what happened, parallel branches and all, and it never alters existing commits. The cost is a messier graph full of merge commits that can be noisy on a busy project. Rebase gives you a clean, linear, easy-to-read history that is pleasant to bisect through later. The cost is that it rewrites commits, so it is dangerous on shared branches, and the linear history is a slight fiction (it pretends work happened sequentially when it did not). A common team compromise: rebase your own feature branch to keep it current and tidy, then merge it into main with --no-ff so the feature stays visible as a unit.

Cherry-pick: grabbing a single commit

Sometimes Sahil does not want a whole branch. Maybe one commit on add-search fixes a typo that is also broken on main right now, and he cannot wait for the whole feature to merge. git cherry-pick copies the changes from one specific commit and applies them as a new commit on his current branch. Like rebase, it creates a new commit with a new hash, because it is the same change applied in a different place, not the same commit moved.

terminal
$ git switch main
$ git log --oneline add-search
e5f6a7b add search keyboard shortcut
d4c3b2a fix typo in error message   <- want just this one
a1b2c3d add search input component

$ git cherry-pick d4c3b2a
[main 9f8e7d6] fix typo in error message
 1 file changed, 1 insertion(+), 1 deletion(-)

Use cherry-pick sparingly. If you cherry-pick a commit now and later merge the whole branch, the same change can show up twice, which sometimes causes conflicts. It is a precision tool for "I need exactly this one fix over here," not a general way to move work around.

git merge <branch>Combine the named branch into your current branch. Fast-forwards if possible, otherwise creates a merge commit.
git merge --no-ff <branch>Force a merge commit even when a fast-forward was possible, keeping the feature visible as a unit.
git rebase mainFrom your feature branch, replay your commits on top of the latest main. Rewrites your commits (new hashes).
git rebase --abortBail out of an in-progress rebase and return your branch to how it was before you started.
git rebase --continueAfter resolving a conflict during a rebase and staging the fix, resume replaying the remaining commits.
git cherry-pick <hash>Apply the changes from one specific commit as a new commit on your current branch.
git merge --abortCancel an in-progress merge that hit conflicts and restore the pre-merge state.
Tip

When you are unsure what a merge or rebase will do to your history, look first with git log --oneline --graph --all. It draws the commit graph in your terminal so you can see the branches and where they point before you change anything.

SEC 04/ RECOVERY

Fixing conflicts & undoing mistakes

I'll write this teaching section. Since the global instructions exempt teaching content from caveman mode and require clear full prose, I'll write accordingly.

Sahil runs a merge, and instead of a clean "all done," Git stops cold and says CONFLICT. This is the moment most people panic. It is also the moment Git is actually being honest with you.

Sahil has been working on a branch called feature-login. While he was busy, a teammate changed the same file on main. Now Sahil tries to merge main into his branch, and Git hits a wall. Here is the thing to understand first: a merge conflict is not Git breaking. It is Git refusing to guess. When two branches change the same lines of the same file in different ways, Git has no way to know which version you actually want, so it stops and hands the decision back to you. That is the correct behavior. The alternative (silently picking one) would quietly destroy work.

terminal
$ git merge main
Auto-merging src/auth.js
CONFLICT (content): Merge conflict in src/auth.js
Automatic merge failed; fix conflicts and then commit the result.

$ git status
On branch feature-login
You have unmerged paths.
  (fix conflicts and run "git commit")
  (use "git merge --abort" to abort the merge)

Unmerged paths:
  (use "git add <file>..." to mark resolution)
        both modified:   src/auth.js

Notice that Git tells you exactly what to do next, right there in the output. Read it. Git is not cryptic here. It says which files conflicted, and it reminds you of both ways out: resolve and commit, or abort entirely.

Reading the conflict markers

When Sahil opens src/auth.js, he finds his code has been rewritten with strange lines of angle brackets and equals signs. These are conflict markers. Git did not corrupt the file. It inserted both versions side by side and asked Sahil to choose. Here is what he sees:

src/auth.js
function login(user) {
<<<<<<< HEAD
  return checkPassword(user, "sha256");
=======
  return checkPassword(user, "bcrypt");
>>>>>>> main
}

Read these markers carefully, because they have a precise meaning. Everything between <<<<<<< HEAD and ======= is the version on your current branch (HEAD, which is Sahil's feature-login). Everything between ======= and >>>>>>> main is the version coming in from the branch you are merging (main). The ======= line is just the divider between the two sides. It is not part of your code.

To resolve the conflict, Sahil decides what the final code should actually be. Maybe he wants his version, maybe theirs, maybe a blend of both. Then he deletes all three marker lines and leaves only the real code he wants to keep. The markers must be gone completely. If even one <<<<<<< survives, you have a broken file and probably a broken build.

src/auth.js (resolved)
function login(user) {
  return checkPassword(user, "bcrypt");
}

The resolve loop

Resolving a conflict is always the same rhythm, no matter how many files are involved. Sahil edits each conflicted file by hand, then tells Git "this one is settled" with git add, and once every file is settled he finishes the merge with a commit. The git add step is what marks a file as resolved. Until you add it, Git still considers the conflict open.

  1. Open each conflicted file and edit it so it contains exactly the code you want, with all conflict markers removed.
  2. Run git add <file> on each file you have fixed. This marks the conflict as resolved.
  3. Check git status until no unmerged paths remain.
  4. Finish with git commit (or git merge --continue, which does the same thing for a merge and pre-fills the message).
terminal
# after editing src/auth.js to remove markers
$ git add src/auth.js
$ git status
On branch feature-login
All conflicts fixed but you are still merging.
  (use "git commit" to conclude merge)

$ git commit
[feature-login 3f9a2c1] Merge branch 'main' into feature-login

Seeing the conflict with git diff

While a merge is in progress, git diff behaves differently than usual. Instead of comparing two simple snapshots, it shows you a combined diff of the conflicted regions, which side each change came from, and what is still unresolved. This is useful when a file has many conflicts and Sahil wants a quick map of what still needs his attention before he commits.

Bailing out safely

Sometimes Sahil looks at the conflict, realizes he is not ready to deal with it, or started the merge from the wrong branch entirely. He does not have to push through. He can abandon the whole operation and return to exactly where he was before he started.

terminal
# bail out of a merge that hit conflicts
$ git merge --abort

# bail out of a rebase that hit conflicts
$ git rebase --abort

Both of these are safe and put your branch back to the state it was in before the merge or rebase began. They are the correct escape hatch when you are mid-conflict and want to step away cleanly. Note that they only work while the operation is still in progress. Once you have committed the merge, you need the undo tools below instead.

Tip

If you almost always want one side of a conflict for a specific file, you can take it wholesale: git checkout --ours path/to/file keeps your current branch's version, and git checkout --theirs path/to/file takes the incoming branch's version. Then git add the file as usual. "Ours" means HEAD (the branch you are on), "theirs" means the branch being merged in. Be careful: this replaces the entire file, not just the conflicting lines.

Undo and recover

Conflicts are one kind of mess. The other kind is "I did something and now I want it gone." Git has a recovery tool for almost every situation, but they are not interchangeable, and a few of them throw work away permanently. The single most important skill here is knowing how dangerous each command is before you run it. Let us go from gentlest to most destructive.

Discard changes you have not committed. Sahil edited a file, hated the result, and wants the file back the way it was at the last commit. git restore <file> throws away the uncommitted edits in that file. This is mildly destructive: the edits are gone, because uncommitted work is the one thing Git cannot recover for you. There is no version of it stored anywhere.

Unstage something you added too early. Sahil ran git add on a file he did not mean to include yet. git restore --staged <file> moves it back out of the staging area without touching the file's contents. This is completely safe. Your edits are untouched; you have only changed what is queued for the next commit.

Move the branch backward with reset. When Sahil wants to undo one or more commits, he uses git reset, and the mode he picks decides what happens to his work. This is where people lose files, so read these three carefully.

git reset --soft HEAD~1Undo the last commit but keep all its changes staged, ready to recommit. Safe. Nothing is lost.
git reset --mixed HEAD~1Undo the last commit and unstage its changes, but keep them in your working files. This is the default. Safe.
git reset --hard HEAD~1Undo the last commit AND delete its changes from your working files entirely. Destructive.
Destructive

git reset --hard permanently discards your uncommitted changes and throws away the commits you reset past. If those changes were never committed, they are gone for good and even git reflog cannot bring them back. Only run --hard when you are certain you want to lose that work. When in doubt, use --soft or --mixed instead, which keep your changes around.

Undo a commit that other people already have. Reset rewrites history by moving the branch pointer, which is fine on your own private branch but dangerous on a shared branch, because everyone else's history will no longer match yours. The safe public undo is git revert. Instead of erasing a commit, it creates a brand new commit that applies the exact opposite changes. History stays intact, nothing is rewritten, and your teammates can pull normally.

terminal
# a08fe21 is the bad commit you want to undo on a shared branch
$ git revert a08fe21
[main 7c1d04e] Revert "Add broken password check"

Fix the commit you just made. Sahil committed, then immediately noticed a typo in the message or a file he forgot to include. git commit --amend replaces the most recent commit with a new one. Stage anything you forgot, run the amend, and the last commit is rebuilt. Be aware that amend rewrites that commit, so if you already pushed it, you have the same shared-history problem as reset. Amend freely on commits you have not pushed.

Save your work to switch context mid-change. Sahil is halfway through editing when an urgent bug comes in on another branch. He is not ready to commit a half-finished change, but he cannot switch branches with a dirty working tree cleanly. git stash tucks all his uncommitted changes away on a shelf and gives him a clean working tree. He fixes the bug, switches back, and runs git stash pop to bring his half-done work back exactly as it was.

git stashSave all uncommitted changes aside and revert your working tree to clean.
git stash listShow all the stashes you have saved.
git stash popReapply the most recent stash and remove it from the stash list.
terminal
$ git stash
Saved working directory and index state WIP on feature-login: 3f9a2c1
$ git switch main
# ... fix the urgent bug, commit it ...
$ git switch feature-login
$ git stash pop
# your half-finished changes are back

The safety net: git reflog

Here is the command that turns most "I destroyed my work" panics into "oh, it's fine." Git keeps a private log of every place HEAD has pointed, including commits you reset past, rebased away, or otherwise "lost." That log is the reflog. As long as a change was committed at some point, the reflog almost certainly still has a reference to it, even after a hard reset.

terminal
$ git reflog
3f9a2c1 HEAD@{0}: reset: moving to HEAD~1
a08fe21 HEAD@{1}: commit: Add broken password check
7c1d04e HEAD@{2}: commit: Add login form

# Sahil sees the commit he thought he lost is a08fe21
$ git reset --hard a08fe21
# or, to inspect it safely first:
$ git checkout a08fe21

So if Sahil ever runs a reset and his heart sinks, his first move is git reflog, not despair. He finds the commit hash from before the mistake and resets or checks it back out. The one thing reflog cannot save is work that was never committed at all, which is exactly why committing often is itself a recovery strategy.

Destructive

git push --force overwrites the branch on the remote with your local version, deleting any commits on the remote that you do not have. On a shared branch this can erase a teammate's work permanently and there is no reflog on their machine to save you from your decision. If you genuinely must force push after an amend or rebase, use git push --force-with-lease instead. It refuses to overwrite if someone else has pushed since you last fetched, which is the guardrail that keeps a force push from silently destroying work you did not know about.

SEC 05/ DISTRIBUTED

Working with remotes: going distributed

Sahil's taskboard works on his laptop, but a project that lives in one place is one spilled coffee away from gone. Time to share it.

So far Sahil has been committing happily on his own machine. The history is real, the branches work, everything is fine. But his repository exists in exactly one location: a folder on his laptop. If the disk dies, the project dies with it. And his teammate Priya cannot see a single line of it. Both problems have the same answer, which is to put a copy of the repository somewhere they can both reach, like GitHub.

The core idea: Git is distributed

Here is the part that trips up people coming from older tools. In Git there is no single "master" copy that everyone connects to. Sahil's local repository is complete and self-sufficient. It has the entire history, every branch, every commit, all stored right there in the .git folder. He can commit, branch, and look at history with the network unplugged.

A "remote" is simply another full copy of that same repository living somewhere else, usually on a server like GitHub or GitLab. By strong convention, the first remote you add is named origin. The name is just a label, a nickname for a URL, so Sahil types origin instead of pasting a long address every time.

Now, the subtle bit. When Sahil connects to origin, Git keeps two kinds of branches in his local repo. There are his local branches (like main), which are the ones he actually commits to and moves around. And there are remote-tracking branches (like origin/main), which are Git's read-only memory of where that branch was on the server the last time he talked to it. Think of origin/main as a bookmark, a snapshot saying "this is where the server's main branch stood when I last checked." It does not update on its own. It only moves when Sahil explicitly fetches or pushes. Keeping these two ideas separate is the single most useful mental model for everything that follows.

git remote: introducing Sahil's repo to the server

Sahil created his repo with git init, which sets up a local repository but does not know about any server. (If he had instead run git clone on an existing project, Git would have set up origin for him automatically. That is the main practical difference between the two.) So he goes to GitHub, creates an empty repository called taskboard, and tells his local repo where it lives.

git remote add origin <url>Register a remote named "origin" pointing at the given URL.
git remote -vList remotes with their fetch and push URLs (verbose).
git remote rename origin upstreamRename a remote.
git remote remove originForget a remote. This does not touch your commits, only the bookmark to the server.
terminal
$ git remote add origin https://github.com/sahil/taskboard.git
$ git remote -v
origin  https://github.com/sahil/taskboard.git (fetch)
origin  https://github.com/sahil/taskboard.git (push)

Nothing has been uploaded yet. All Sahil did was write down an address. The commits are still only on his laptop.

git push: sending commits up, and linking the branches

Now Sahil actually sends his work to the server. The first push is slightly special. He uses -u (the short form of --set-upstream) to do two things at once: upload his main branch, and permanently link his local main to origin/main.

terminal
# First push: upload main AND set up tracking
$ git push -u origin main
Enumerating objects: 12, done.
To https://github.com/sahil/taskboard.git
 * [new branch]      main -> main
branch 'main' set up to track 'origin/main'.

# Every push after this can be just:
$ git push

That tracking link is what "upstream" means, and it is pure convenience. Once main tracks origin/main, Sahil can type a bare git push and Git knows where to send it. He also gets handy status lines like "Your branch is ahead of origin/main by 2 commits" when he runs git status, because Git can now compare his local branch against its remote-tracking bookmark. Without -u, he would have to spell out git push origin main every single time.

git push -u origin mainFirst push of a branch: upload it and set upstream tracking.
git pushPush the current branch to its tracked upstream (after -u is set once).
git push origin mainExplicitly push a named branch to a named remote.

git fetch vs git pull: the difference that bites beginners

A few days later Priya has pushed her own commits to origin. Sahil wants them. There are two commands here, and confusing them is one of the most common sources of beginner panic, so be precise about what each one does.

git fetch downloads new commits from the server and updates Sahil's remote-tracking branches (origin/main moves forward). It does not touch his working files and does not move his local main. It is completely safe: nothing he is editing changes, no merge happens, no conflict can occur. After a fetch, Sahil can inspect what arrived before deciding what to do with it.

git pull is two commands glued together: it runs git fetch and then immediately git merge to integrate those new commits into his current branch. This does change his working files, and if Priya edited the same lines he did, this is the moment a merge conflict appears. With --rebase, the second step becomes git rebase instead of git merge, which replays his local commits on top of Priya's for a cleaner, straight-line history.

git fetch originDownload new commits and update origin/* pointers. Working files untouched.
git pullfetch + merge into the current branch. Changes your files.
git pull --rebasefetch + rebase your local commits on top. Keeps history linear.
git log --oneline origin/mainInspect what you fetched before integrating it.
terminal
# SAFE: see what's new without changing anything
$ git fetch origin
From https://github.com/sahil/taskboard
   a1b2c3d..e4f5g6h  main -> origin/main

$ git status
Your branch is behind 'origin/main' by 1 commit, and can be fast-forwarded.

# Now look at exactly what Priya did before touching your files
$ git log --oneline main..origin/main
e4f5g6h Add filter buttons to task list

# Happy with it? THEN integrate. This is what pull would have done in one step.
$ git merge origin/main
Updating a1b2c3d..e4f5g6h
Fast-forward
Tip

When in doubt, fetch first. git fetch can never lose or scramble your work, because it only updates the read-only origin/* bookmarks and leaves your files and local branches exactly as they were. git pull is just fetch with an automatic integration step bolted on. Splitting them lets you look before you leap.

The "rejected, non-fast-forward" error

Here is the scenario that will eventually happen to Sahil. He made a commit locally. Meanwhile, Priya pushed a commit to origin/main that he does not have yet. Sahil runs git push and Git refuses:

terminal
$ git push
To https://github.com/sahil/taskboard.git
 ! [rejected]        main -> main (non-fast-forward)
error: failed to push some refs to 'https://github.com/sahil/taskboard.git'
hint: Updates were rejected because the remote contains work that you do
hint: not have locally. Integrate the remote changes (e.g. 'git pull ...')
hint: before pushing again.

This is not a bug, it is Git protecting shared work. The server's main has moved ahead of where Sahil thinks it is, so accepting his push would mean throwing away Priya's commit. Git will not silently do that. The fix is exactly what the hint says: bring her work into your branch first, then push.

terminal
# Integrate Priya's commit, replaying yours on top for a clean line
$ git pull --rebase
From https://github.com/sahil/taskboard
   a1b2c3d..e4f5g6h  main -> origin/main
Successfully rebased and updated refs/heads/main.

# Now your branch builds ON TOP of hers, so the push is a fast-forward
$ git push
To https://github.com/sahil/taskboard.git
   e4f5g6h..9z8y7x6  main -> main

If the rebase or merge surfaces a conflict because you both edited the same lines, Git pauses and asks you to resolve it. That is normal collaboration, not an error, and the conflicts section covers resolving them in detail.

Destructive

You may be tempted to "fix" a rejected push with git push --force. On a shared branch this is genuinely dangerous: it overwrites whatever is on the server with your version, permanently deleting any commits (like Priya's) that you do not have locally. Her work is simply gone from the server, and she will get errors on her next pull. Never force-push a branch other people use. If you absolutely must rewrite a branch only you own, use git push --force-with-lease instead. It refuses to overwrite if the remote has moved since you last fetched, so it cannot silently clobber commits you have not seen.

A typical day in the loop

Once the model clicks, Sahil's daily rhythm with the server settles into a small, predictable loop. Pull first so he is building on the latest shared work, then do his actual changes, commit them locally, and push them up for Priya to see.

terminal
# 1. Start the day on top of everyone else's work
$ git pull --rebase

# 2. Do the work, then stage and commit it locally
$ git add .
$ git commit -m "Add due-date field to tasks"

# 3. Share it
$ git push

That is the whole heartbeat of distributed work: integrate, build, record, share. Master this loop and remotes stop feeling mysterious. They are just other copies of your repository that you periodically sync with, on your terms, with commands whose effects you can now predict.

SEC 06/ COLLABORATION

Pull Requests: review & merge

Here is the inner HTML for the section body.

A Pull Request is not a Git command. It is a conversation built on top of a push.

Sahil has finished his work on a branch called feature/login-form. The commits are good, the branch is pushed to the remote, and the code works on his machine. In a solo project he would just merge it into main and move on. But Sahil is on a team now, and the team has a rule: nothing reaches main without another person looking at it first. This is where a Pull Request comes in.

A Pull Request (called a Merge Request on GitLab, the same idea under a different name) is a formal request to merge one branch into another, usually your feature branch into main. The important thing to understand is that the PR itself is a feature of the hosting platform, GitHub, GitLab, or Bitbucket, and not part of Git itself. Git knows about commits, branches, and remotes. It has no concept of a "review" or an "approval." The platform wraps a branch comparison in a web page that adds discussion, line-by-line comments, automated checks, and an approval workflow. When you click the merge button, the platform runs an ordinary git merge on the server for you.

Fork model versus branch model

There are two ways to get your code into a position where a PR can be opened, and which one you use depends on whether you can write to the repository directly. In the branch model, Sahil has push access to the team's repository, so he pushes a branch straight to it and opens the PR from that branch. This is normal for company teams and private projects. In the fork model, Sahil does not have write access (this is the usual situation when contributing to someone else's open-source project), so he first creates a fork, which is his own server-side copy of the repository, pushes his branch to the fork, and opens the PR from his fork into the original repository. The review and merge experience is identical. The only difference is where the branch physically lives.

The lifecycle, step by step

Here is the full journey Sahil's change takes, from finished commits to merged code. Each step has a clear purpose, so it helps to know why it exists and not just the order.

  1. Sahil pushes his branch to the remote with git push -u origin feature/login-form. Nothing is reviewed yet. He has only made his commits visible to others.
  2. He opens a Pull Request on the platform, choosing his branch as the source and main as the target. He writes a title and a description explaining what changed and why.
  3. Continuous Integration (CI) checks fire automatically. These are scripts the team configured to run tests, linters, and builds against the PR. If they fail, the PR is marked with a red status and usually cannot be merged until they pass.
  4. A teammate reviews the diff and leaves comments. These can be general remarks or comments pinned to specific lines of code.
  5. If the reviewer wants changes, they formally "request changes," which blocks the merge until resolved. Sahil does not open a new PR. He simply makes more commits on the same branch and pushes again.
  6. Because the PR tracks the branch, the new commits appear in the same PR automatically and CI re-runs. The review continues on the updated code.
  7. Once satisfied, the reviewer approves the PR. With approvals in place and CI green, the merge button unlocks.
  8. The PR is merged into main using one of the three merge strategies below, and the feature branch can then be deleted.

The three merge strategies

When the merge button is finally pressed, the platform offers up to three ways to fold the branch into main. They produce genuinely different history, so the choice matters.

A merge commit keeps every commit from the branch exactly as it was and adds one extra commit that ties the two histories together. This preserves the true sequence of what happened, but if Sahil made ten messy "fix typo" commits, all ten land in main and clutter the log. A squash merge collapses every commit on the branch into a single new commit on main. The history stays clean and each PR becomes one tidy entry, but the granular step-by-step commits are lost. A rebase merge replays each branch commit one by one directly onto the tip of main with no merge commit, producing a perfectly linear history, but it rewrites the commit hashes and loses the record that these commits ever lived on a separate branch.

Watch out

Squash and rebase merges rewrite history. The commits that end up on main are brand new objects with different hashes than the ones on Sahil's branch. After the merge he should delete his local feature branch and pull main fresh rather than trying to reuse or re-merge the old branch, because Git no longer sees those commits as the same.

Doing it from the terminal with the gh CLI

Sahil does not have to leave the command line to open or check out a PR. GitHub ships a tool called gh that talks to the platform's API. With it he can create a PR straight after pushing, and a teammate can pull down Sahil's PR branch by its number to test it locally without manually figuring out the remote and branch names.

terminal
$ git push -u origin feature/login-form
# branch pushed and tracking set up

$ gh pr create --base main --head feature/login-form --title "Add login form" --body "Implements the sign-in screen"
# opens the PR and prints its URL
https://github.com/acme/webapp/pull/42

# a teammate checks out the PR locally to test it
$ gh pr checkout 42
Switched to branch 'feature/login-form'

$ gh pr status
# shows the PR's checks, reviews, and merge state
git push -u origin <branch>Push your branch to the remote and set it to track, the prerequisite for opening a PR.
gh pr createOpen a Pull Request from the current (or specified) branch into a target branch.
gh pr checkout <number>Check out someone else's PR branch locally by its PR number to test it.
gh pr statusShow the state of the PR: CI checks, reviews, and whether it can merge.
git fetch && git pull origin mainAfter the PR merges, refresh your local main so it includes the merged work.
git branch -d <branch>Delete the now-merged local feature branch to keep things tidy.
Tip

Open the PR early, even before the work is finished, as a Draft Pull Request. It lets CI run on every push and gives teammates a place to comment on direction before Sahil has poured hours into the wrong approach. You mark it ready for review only once it is actually done.

SEC 07/ DISCIPLINE

Best practices: keeping history sane

I'll write this section following the teaching style rules. Let me produce the inner HTML directly.

A year from now, Sahil will open this repo and read its history like a story. Whether that story makes sense is decided today, one habit at a time.

When Sahil's team started their app, history was a junk drawer. Commits like "stuff", "fix", and "wip again" piled up. Six months later nobody could answer a simple question: when did the login bug get introduced, and why? The fix was not a tool. It was a set of conventions everyone agreed to follow. None of these are enforced by Git itself, which is exactly why they matter. Git happily lets you do the wrong thing forever.

Atomic commits: one logical change each

An atomic commit captures exactly one logical change and nothing else. When Sahil renames a function and also fixes an unrelated typo in the footer, those belong in two separate commits. The reason is not tidiness for its own sake. It is that git revert, git cherry-pick, and git bisect all operate on whole commits. If the rename and the typo fix are glued together, Sahil cannot undo one without dragging the other along. Use git add -p to stage selected hunks when you accidentally mixed two changes in your working tree, so each commit stays focused.

Tip

A good test: can you describe the commit in one sentence without using the word "and"? If you need "and", it is probably two commits.

Commit messages that still make sense later

A commit message is a note to a future stranger, and that stranger is usually you. The convention most teams follow is the 50/72 rule. The subject line stays at or under 50 characters, written in the imperative mood, as if completing the sentence "If applied, this commit will...". So Sahil writes "Add password reset endpoint", not "Added password reset" and not "Adding password reset". This matches the voice Git itself uses for its automated messages like "Merge branch".

If the change needs explanation, leave a blank line after the subject, then write a body wrapped at around 72 characters. The body is where the why lives. The diff already shows what changed. What it cannot show is why Sahil chose this approach, what he ruled out, or which bug report drove it. That context is what saves an hour of archaeology next year.

Many teams formalize this with Conventional Commits, where the subject starts with a type, such as feat:, fix:, docs:, refactor:, or chore:. This is not just decoration. Tooling can read those prefixes to generate changelogs and bump version numbers automatically, so a feat: bumps a minor version and a fix: bumps a patch.

terminal
$ git commit
# Editor opens. Sahil writes:

feat: add rate limiting to login endpoint

Brute-force attempts were not throttled, so an attacker could
guess passwords as fast as the network allowed. This caps each
IP at 5 attempts per minute using the existing Redis store.

Closes #142

Commit early and often, but clean the noise before sharing

These two ideas seem to contradict each other, but they do not. While Sahil is working locally, he should commit constantly, even messy "checkpoint" commits, because a commit is a save point he can return to. The freedom to experiment safely comes from committing often. The catch is that this messy local history should not become everyone else's problem. Before he pushes a branch for review, Sahil tidies it with an interactive rebase, squashing five "wip" commits into one or two clean, atomic ones.

Destructive

Rebasing and squashing rewrite history by creating new commits with new hashes and discarding the old ones. Only ever rewrite commits that live solely on your own un-shared branch. Rewriting commits others have already pulled forces them into a painful manual recovery, and on a shared branch a git push --force can silently erase a teammate's work. Use git push --force-with-lease instead, which refuses to push if someone else has updated the branch since you last fetched.

Never commit secrets or large binaries

This is the rule with the longest blast radius. The moment Sahil commits an API key or a database password, it is in the history permanently. Deleting it in a later commit does nothing, because the secret still sits in every clone of the repo, recoverable from the old commit. The only real fix is to rotate the leaked credential (assume it is now public) and then rewrite history with a tool like git filter-repo, which is slow and disruptive for everyone. Prevention is enormously cheaper than the cure.

Prevention has two layers. First, a .gitignore file that lists things Git should never track, such as .env files, node_modules/, build output, and large media. Second, a guard like the git-secrets tool, which installs a pre-commit hook that scans staged changes for patterns that look like keys and blocks the commit before the mistake can happen. Large binaries deserve their own warning: Git stores a full new copy of a binary on every change (it cannot diff them efficiently), so committing big files bloats the repo permanently for everyone who clones it. Reach for Git LFS if you genuinely need to version large assets.

Watch out

Adding a secret to .gitignore after you have already committed it does nothing. The file is already tracked, so Git keeps watching it. You must untrack it with git rm --cached <file>, and you must still rotate the leaked secret because it remains in history.

Pull with rebase to avoid noisy merge bubbles

When Sahil runs a plain git pull and his local branch has diverged from the remote, Git creates an extra "merge commit" just to stitch the two together. Do that a few dozen times and the history fills with meaningless "Merge branch main into main" bubbles that drown out the real work. Pulling with rebase instead replays Sahil's local commits on top of the freshly fetched ones, producing a clean, linear history with no junk merge commit. He can set this as the default so he never has to remember.

Short-lived branches and a deployable main

The longer a branch lives, the further it drifts from main, and the more painful the eventual merge becomes. Sahil's team keeps feature branches small and short-lived, ideally merged within a day or two, so conflicts stay tiny. The companion rule is that main is always deployable. Nobody pushes half-finished work directly to main; broken or experimental code stays on a branch. This way, if production catches fire, the team can ship from main at any moment with confidence.

Review before merge, and tag your releases

Every change reaching main goes through review via a pull request. Review is not about distrust. It is a second pair of eyes that catches bugs early and spreads knowledge of the codebase so no single person becomes a bottleneck. Finally, when the team ships a real release, they mark it with an annotated tag. An annotated tag (created with -a) is a full object that stores a message, the tagger, and a date, unlike a lightweight tag which is just a bare pointer. That metadata is what lets Sahil answer "what exactly did we ship as v1.2.0 last quarter?" without guessing.
